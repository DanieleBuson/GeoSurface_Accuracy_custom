# THEORY — Surface confidence assessment & stratigraphic boundaries and faults overlap for 3D geological models

This document explains **what the pipeline in this repository computes, why, and what every
generated file means**. It is self-contained: you do not need access to any other repository
to understand it.

## 1. Where this comes from

This pipeline is a dataset-specific application of the **GeoSurface_Accuracy** tool
(originally [BaterHub/GeoSurface_Accuracy](https://github.com/BaterHub/GeoSurface_Accuracy)),
which estimates *confidence maps* for geological surfaces stored in GOCAD `.ts` format —
i.e. "how much do we trust the modeled surface elevation/position at each location?",
based on how close that location is to actual control data (wells, sections, mapped
contacts, topography intersections).

All reading/IDW/plotting logic lives in [`files_utils.py`](./files_utils.py) (a verbatim
copy of the upstream tool's implementation, vendored here so this folder can be cloned and
run on its own). [`custom_utils.py`](./custom_utils.py) is just a re-export of
`files_utils.py`, kept so the existing notebooks/scripts don't need to change their imports.
[`custom_validation.py`](./custom_validation.py) holds the dataset-specific extensions:
corrected stratigraphic contact matching, the symmetric boundary-overlap metric, fault
validation, fault-throw-impact layer per horizon, and acceptance classification.

Compared to the original tool's reference dataset (which has wells + seismic sections),
**this dataset has no wells**, but adds several control sources the original tool didn't
use: explicitly-mapped geological contacts and faults (`Limiti_stratigrafici_carta_geologica_DEF.shp`,
`Faglie_carta_geologica_DEF.shp`), software-computed topography intersections for both
horizons and faults (`3D_Topo_Intersections*.shp`), and a dedicated fault-surface model
(`GOCAD_ASCII_Faults.ts`). Supporting those required:

- a **"maps" horizontal-confidence tier** (geological-map contact lines, `order_p=3`,
  weighted by field-reliability),
- a **symmetric (bidirectional) boundary-overlap metric** comparing the model's computed
  outcrop trace against the independently-mapped contact, and
- an equivalent **fault validation** metric comparing modeled fault traces against
  GIS-mapped faults, using the same symmetric methodology.

## 2. The dataset

All inputs live in [`working_files_folder/`](./working_files_folder/) (this pipeline reads
the `original/` copies of the GOCAD files — full, complete horizons/faults, never
split/segmented/partial surfaces):

| File | Content |
| --- | --- |
| `GOCAD_ASCII_All.ts` (read from `original/`) | 7 geological surfaces (`TOP_ARV`, `TOP_DPR`, `TOP_LOP`, `TOP_RTZinf`, `TOP_FMZ`, `TOP_RTZsup`, `TOP_OSV`) of a regional 3D geological model, 343,861 vertices total |
| `GOCAD_ASCII_Faults.ts` (read from `original/`) | Fault surfaces used for fault validation |
| `3D_SectionsGrid.shp` | Control sections — 22 generic 3D section-trace lines (field `Name`), used as a "sections" checkpoint source for **every** surface |
| `3D_Topo_Intersections.shp` | Horizon-topography intersections generated by MOVE — per surface, the 3D line where that surface **as computed by the modeling software** crops out at the topography (field `Horizon` = `TOP_ARV`, `TOP_DPR`, ...). Geometry Z = DEM elevation. |
| `3D_Topo_Intersections_Faults.shp` | Fault-topography intersections generated by MOVE (field `Name`, e.g. `F8a`) |
| `Limiti_stratigrafici_carta_geologica_DEF.shp` | Stratigraphic boundaries mapped in GIS — **the sole source of cartographic stratigraphic-contact reliability and geometry** (`Base_di`, `Tipo`, `Affidabili`) |
| `Faglie_carta_geologica_DEF.shp` | Faults mapped in GIS (`Nome_fagli`, `Tipo_fagli`) — **the official GIS reference for all fault validation** |

`Limiti_CartaGeol.shp` is **fully retired**:
`Limiti_stratigrafici_carta_geologica_DEF.shp` replaces it everywhere in this pipeline —
confidence maps, boundary overlap, cartographic checks, and statistics. There is no legacy
fallback path; if this file is missing, the maps tier and boundary-overlap metric are simply
empty for every surface.

There are **no well shapefiles** in this dataset, so `use_wells=False` everywhere.

There is a single run (`run_accuracy_original.py` → `output_results_original/`) over the
full original horizon/fault surfaces from `original/GOCAD_ASCII_All.ts` /
`original/GOCAD_ASCII_Faults.ts` — confidence-map processing must never run on
split/segmented/partial surfaces. An earlier version of this pipeline also ran a
"resampled" variant over densified topo-intersection shapefiles (0.1 m vertex step) purely
as an IDW-result consistency check; it has since been removed as redundant.

## 3. Coordinate reference system

All shapefiles and the `.ts` model use the same projected meters, standardized to
**`EPSG:6707`** (RDN2008 / UTM zone 32N) right after loading via `files_utils.standardize_crs`
— including layers whose `.prj` originally declared a different (but numerically identical)
UTM32N authority code, and `3D_Topo_Intersections_Faults*.shp`, which carries no recognized
authority code at all. No layer is exempt; this is the single CRS baseline for the whole
pipeline.

Per the "Linee Guida" PDF (§5, production workflow step 1, see §7 below), the **national
GeoIT3D submission standard** re-projects models to **`EPSG:6708` (RDN2008 / UTM zone
33N)**. This dataset is **not** reprojected here — this is a local accuracy test, not a
submission package — but anyone preparing a real GeoIT3D delivery from this model would
need that reprojection step.

## 4. Method

### 4.1 Grid + hull
For each surface, build a 2D evaluation grid at a fixed spacing (`GRID_SPACING`, 200 m in
both runs), clipped to the convex hull of that surface's footprint
(`build_grid` in `files_utils.py`).

### 4.2 Sample controls
- **Sections** (`3D_SectionsGrid.shp`): resampled along their geometry every
  `LINE_STEP` (200 m) to get dense points (`sample_lines_gdf`).
- **Maps** (`Limiti_stratigrafici_carta_geologica_DEF.shp`): for each surface, the subset of
  contacts whose `Base_di` matches that surface (see §5) is resampled every `MAPS_STEP`
  (100 m).
- **Wells**: not present in this dataset (`use_wells=False`).

### 4.3 Horizontal confidence (IDW)
For each grid node, compute the nearest distance to each available control type
(sections, maps). Convert distance `r` (km) to an inverse-distance score
`ID = 1 / (1 + r^p)`, then min–max normalize per type to `[0, 1]`
(`compute_order_weight`). The exponent `p` differs by control type — **sections use
`p=2`, maps use `p=3`** (wells would use `p=1` if present) — so point-like, higher-order
controls (maps) decay faster with distance than line-like, lower-order controls
(sections).

For the **maps** tier specifically, each sampled point's weight is additionally
multiplied by a **reliability factor** derived from that contact's `Affidabili` field
(`AFFIDABILI_WEIGHTS`, see §5.2) — the **only** source of cartographic reliability used
anywhere in this pipeline — and re-clipped to `[0, 1]`. This means a grid node close to a
contact marked `Ipotizzato` ("hypothesized") gets less horizontal-confidence credit than the
same distance to a contact marked `Osservato` ("observed").

The final `weight_combined` is the **mean of the available per-type normalized weights**
at each grid node (`compute_horizontal_weights`). A ranking plot sorts all grid nodes by
descending `weight_combined` — since `weight_combined` already folds in the `Affidabili`
weighting, the ranking plot reflects cartographic reliability as well as distance.

### 4.4 Vertical confidence (|ΔZ|)
Wherever checkpoints carry a Z value, the surface's interpolated elevation at that XY
(via Delaunay/linear triangulation of the surface vertices, `LinearNDInterpolator`) is
compared against the checkpoint's Z: `delta_z = z_checkpoint - z_surface`. The absolute
residual `|delta_z|` is IDW-interpolated onto the same evaluation grid
(`idw_power=2`), giving `abs_delta_z`. A normalized version is also produced:
`abs_delta_norm = 1 - (|dZ| - min)/(max - min)`, so **1 = best** (smallest residual) and
**0 = worst** (largest residual) on that surface's grid.

In this dataset, the **only source of Z checkpoints is `3D_Topo_Intersections*`**
(no wells, no depth CSVs, no section Z) — see §5.3. Geometry Z in that shapefile is the
**DEM elevation** at the point where the modeled surface crops out at the topography,
i.e. the most direct ground truth available: if the 3D model is correct, the surface's
elevation at that XY *should* equal the DEM elevation there.

### 4.5 Combined confidence (H + V)
Where both a horizontal weight `H` and a normalized vertical confidence `V` exist on the
same grid, three combinations are computed (`generate_combined_confidence`), all driven by
the same `Affidabili`-weighted `H` and topo-derived `V`:
- **arithmetic**: `alpha*H + (1-alpha)*V`
- **geometric**: `H^alpha * V^(1-alpha)`
- **min**: `min(H, V)`

Both run scripts use `alpha=0.5` and select **`mode="min"`** for the headline
PNG/HTML — i.e. *"this location is only as trustworthy as its weakest dimension"*. The
geometric variant is always also saved as a PNG for comparison.

### 4.6 Boundary overlap (model vs. mapped trace) — symmetric, bidirectional metric
This dataset offers **two independent representations of each surface's outcrop trace**:

- **`3D_Topo_Intersections`** (line **A**) — the trace **interpolated by the modeling
  software** (surface ∩ DEM): "what the 3D model thinks the outcrop is".
- **`Limiti_stratigrafici_carta_geologica_DEF`** (line **B**, non-tectonic contacts only) —
  the trace **explicitly mapped in the field/GIS**, independent of the 3D model: "what the
  geologist mapped on the ground".

Before comparison, every segment belonging to the same horizon is merged into a single
geometry on each side via **dissolve + line-merge** (`linemerge(unary_union(...))` in
`generate_enhanced_boundary_overlap`) — e.g. all segments of `TOP_ARV`'s topo-intersection
become one line A, and all of its matched GIS contacts become one line B. Statistics are
always computed on these two whole-horizon geometries, never on individual segments.

The metric is then computed in **both directions**, each using the same fixed tolerance
(`BUFFER_DIST_M = 20 m`):

```
overlap_pct_A_to_B = 100 * length(intersection(A, buffer(B, 20 m))) / length(A)
overlap_pct_B_to_A = 100 * length(intersection(B, buffer(A, 20 m))) / length(B)
overlap_pct_mean   = (overlap_pct_A_to_B + overlap_pct_B_to_A) / 2
```

**`overlap_pct_mean` is the official comparison value.** Computing both directions and
averaging avoids the asymmetry of the old one-directional metric, where a surface could
score near 100% simply because the topo trace was much shorter than (and fully contained
within a buffer around) a much longer mapped contact, or vice versa — see §10. It is a
purely 2D planar comparison: both linesets are projected to 2D (Z dropped) before
buffering/length calculations.

### 4.7 Fault validation — same symmetric methodology, applied per fault

Fault validation follows **exactly the same logic as horizon boundary overlap** (§4.6), so
results are directly comparable between horizons and faults:

- **Model**: `GOCAD_ASCII_Faults.ts` (fault surfaces) and their topo-intersections,
  `3D_Topo_Intersections_Faults*.shp` (line **A** per fault, field `Name`, e.g. `F8a`).
- **GIS**: `Faglie_carta_geologica_DEF.shp` (line **B** per fault, field `Nome_fagli`, e.g.
  `8a`), the official GIS reference for all fault validation.

Before comparison, all segments sharing the same fault identifier are dissolved into one
geometry per side via `dissolve_lines_by_key` (e.g. all `Nome_fagli == '8a'` rows become one
merged GIS line for fault `8a`; all MOVE segments named `F8a` become one merged MOVE line).
The two id systems are normalized to a common key (`_normalize_fault_id`: `'F8a'` and
`'8a'` both become `'8A'`) so the same fault is matched on both sides. Faults present in
only one source are skipped (never guessed) and logged. Statistics are computed on the full
dissolved fault geometry, never on individual segments — covering simple ids (`1`, `2`, ...,
`10`) as well as split fault strands (`8a`/`8b`, `9a`/`9b`).

For each matched fault, using the same 20 m buffer as horizons:

```
fault_overlap_A_to_B = 100 * length(intersection(A, buffer(B, 20 m))) / length(A)
fault_overlap_B_to_A = 100 * length(intersection(B, buffer(A, 20 m))) / length(B)
fault_overlap_mean   = (fault_overlap_A_to_B + fault_overlap_B_to_A) / 2
```

`fault_overlap_mean` is the main fault-validation parameter. **No P95 or other
distance-based metric is used anywhere in fault validation** (or anywhere else in this
pipeline) — `*_overlap_mean` is the only surviving comparison value, for both horizons and
faults.

## 5. How the general method maps onto this dataset

| Concept | This dataset |
| --- | --- |
| Wells (`p=1`) | Not present — `use_wells=False` for all surfaces |
| Sections (`p=2`) | `3D_SectionsGrid.shp`, applied unfiltered to **all 7 surfaces** (it's a generic section grid, not surface-specific) |
| Maps (`p=3`) | `Limiti_stratigrafici_carta_geologica_DEF.shp`, non-tectonic contacts matched to a surface via `Base_di` → `STRAT_MAP` → `_surface_short_code(surface)` (see `select_map_lines_strat`), each sampled point weighted by `Affidabili` reliability (`AFFIDABILI_WEIGHTS`) — the sole source of cartographic reliability |
| Vertical Z checkpoints | `3D_Topo_Intersections*.shp`, matched via `Horizon` → `_normalize_horizon_code`. Geometry Z is the DEM elevation at the modeled outcrop — the most direct ground truth available |
| Fault traces (model) | `GOCAD_ASCII_Faults.ts` + `3D_Topo_Intersections_Faults*.shp`, matched via `Name` |
| Fault traces (GIS) | `Faglie_carta_geologica_DEF.shp`, matched via `Nome_fagli` — the official GIS reference for fault validation |
| Mapping CSVs (`surface_data_mapping.csv` / `surface_checkpoint_edges.csv`) | **Not used.** Maps/topo/fault matching is fully automatic via short codes and the STRAT_MAP translation table; there's nothing to configure per surface |

### 5.1 Surface ↔ contact/horizon matching (`_surface_short_code` & friends)

Both the maps tier and the vertical-checkpoint tier need to know *which*
`Limiti_stratigrafici_carta_geologica_DEF` / `3D_Topo_Intersections` features belong to
*which* of the 7 `.ts` surfaces. This is done by string-matching short codes
(`files_utils.py`) combined with a stratigraphic translation table
(`custom_validation.STRAT_MAP`):

- `_surface_short_code("TOP_RTZinf_merged_resampled_original")` → strips the leading
  `TOP_`/`TOP`, drops everything from `_merged`/`_resampled`/`_split`/`_original` onward,
  and removes non-alphanumerics → `"RTZINF"`. The same function also accepts the cleaned
  `<surface>` form used in output filenames (e.g. `"TOP_RTZinf_original"`) and returns the
  identical `"RTZINF"` code, since matching always happens on the underlying formation
  code regardless of which processing tags are present.
- `_normalize_map_code("Base RTZ inf")` → strips a leading `Base`, removes
  non-alphanumerics → `"RTZINF"`.
- `_normalize_horizon_code("TOP_RTZinf")` → strips a leading `TOP_`/`TOP `, removes
  non-alphanumerics → `"RTZINF"`.

A geological map contact is labelled in the field as "**Base of** the OVERLYING unit" (e.g.
`Base RTZ inf`), while the GOCAD model stores the corresponding surface as "**Top of** the
UNDERLYING unit". `select_map_lines_strat` (`custom_validation.py`) translates between the
two via `STRAT_MAP` before matching, and additionally **excludes tectonic contacts** (rows
where `Tipo` contains "tettonic" but not "non tettonic") — only stratigraphic/non-tectonic
contacts represent a surface's outcrop boundary.

### 5.2 Reliability weighting (`AFFIDABILI_WEIGHTS`)

```python
AFFIDABILI_WEIGHTS = {
    'osservato':  1.00,   # Observed — directly observed in the field
    'dedotto':    0.75,   # Inferred — deduced
    'ipotizzato': 0.50,   # Hypothesized
    'coperto':    0.50,   # Covered (not directly visible)
}
AFFIDABILI_DEFAULT_WEIGHT = 0.75  # Default — missing/unknown Affidabili value
```

`Affidabili`, read exclusively from `Limiti_stratigrafici_carta_geologica_DEF.shp`, is the
**only** source of cartographic reliability anywhere in this pipeline. It drives the
maps-tier weighting that feeds into Horizontal Confidence (§4.3), Combined Confidence
(§4.5), the horizontal-confidence ranking plot, and any other statistic derived from GIS
stratigraphic boundaries.

This is this dataset's local analogue of the ISPRA "Linee Guida" §3.2
`evaluation_method`/`observation_method` controlled vocabulary (see §7): a contact that
was directly observed in the field is worth full reliability, while a hypothesized or
covered contact contributes only half as much to the maps-tier confidence at nearby grid
nodes.

### 5.3 Vertical checkpoints come only from topo intersections

`generate_vertical_outputs` looks for Z in this priority order: well depth CSV → well
geometry, section depth CSV → section geometry, then `3D_Topo_Intersections*` geometry Z.
In this dataset, the first two sources are empty (no wells, no depth CSVs, sections have
no Z), so **every vertical checkpoint comes from `3D_Topo_Intersections*`** — each vertex
of the surface's matched topo-intersection line(s) becomes one `(x, y, z_DEM)` sample.
Every vertex of the matched topo-intersection line(s) becomes one `(x, y, z_DEM)` sample,
so vertical-checkpoint density follows directly from how densely MOVE exported that line.

## 7. Relevant sections of the "Linee Guida" PDF

[`Petricca_etal2025_LineeGuida_StrutturaBD_Nazionale_ModGeo3D.pdf`](./Petricca_etal2025_LineeGuida_StrutturaBD_Nazionale_ModGeo3D.pdf)
(ISPRA, PNRR GeoSciencesIR) defines the national 3D geological model database structure.
The parts most relevant here:

- **§3.2, Tabella 3 (`evaluation_method`)** and **Tabella 4 (`observation_method`)** define
  controlled vocabularies for *how* a surface/contact was determined (compilation, direct
  observation, indirect method, inferred, surveyed; resistivity/seismic/borehole/outcrop
  surveys, etc.). The `Affidabili` field in `Limiti_stratigrafici_carta_geologica_DEF` is
  this dataset's local analogue, and drives the maps-tier reliability weighting (§5.2).
- **§5 (production workflow)** specifies re-projection to `EPSG:6708` (RDN2008/UTM33N)
  for the national submission package (§3 above).
- **§4.4 ("Tool a supporto della valutazione dell'accuratezza")** explicitly points at the
  GeoSurface_Accuracy tool as ISPRA's reference implementation for evaluating the accuracy
  of 3D model surfaces before submission — i.e. this pipeline extends the very tool the
  guidelines describe.

## 9. Output file reference

Every output is written per-surface, with `<surface>` being a cleaned formation name
derived from the `.ts` surface name (`clean_formation_name` in `files_utils.py`):
modeling-software processing tags (`merged`/`resampled`/`split`/`original`) are stripped
from the raw `.ts` name (e.g. `TOP_ARV_merged_resampledoriginal` → `TOP_ARV`), and a single
`_original` suffix is then appended to mark the complete, non-split surface this pipeline
always uses (e.g. `TOP_ARV_original`) — except for `vertical_confidence_grid_<surface>`,
which uses the bare formation name with no suffix (e.g. `TOP_ARV`). Several project-level
outputs (`model_dataset.png`, `boundary_overlap_summary.csv`,
`fault_validation_per_fault.csv`, `fault_validation_aggregate.csv`, `acceptance_table.csv`,
`validation_summary.csv`) are also written once per run.

### 9.1 Whole-model

- **`model_dataset.png`** — footprint of the *entire* model (all 7 surfaces' vertices
  combined), drawn as a bounding-box rectangle over the shared X/Y extent, with the
  coordinate system noted and a scale bar. Gives a single "where is everything" overview.

### 9.2 Per-surface footprint

- **`model_dataset_<surface>.png`** — same style as above but for one surface only, with
  that surface's **evaluation grid points** overlaid (gray dots) so you can see the
  density/coverage of the grid used for all the confidence calculations below.

### 9.3 Horizontal confidence

- **`horizontal_confidence_grid_<surface>.csv`** — one row per evaluation-grid node
  (inside the surface's convex hull). Columns:
  - `x`, `y` — grid-node coordinates in `EPSG:6707` (projected meters).
  - `lon`, `lat` — same point reprojected to `EPSG:4326` (WGS84), for mapping.
  - `dist_wells`, `dist_wells_km`, `weight_wells` — distance to nearest well and its
    normalized IDW weight; **always empty in this dataset** (no wells).
  - `dist_sections`, `dist_sections_km`, `weight_sections` — distance (m / km) to the
    nearest sampled point on `3D_SectionsGrid.shp`, and its normalized IDW weight
    (`order_p=2`).
  - `dist_maps`, `dist_maps_km`, `weight_maps` — distance (m / km) to the nearest sampled
    point on the surface's matched `Limiti_stratigrafici_carta_geologica_DEF` contacts, and
    its normalized, `Affidabili`-reliability-scaled IDW weight (`order_p=3`); empty for
    surfaces with no `Base_di` match (see §5.1).
  - `weight_combined` — mean of the available `weight_*` columns at this node; this is
    **"horizontal confidence", 0–1, higher = better**. This is the value plotted in the
    heatmap and interactive map below.

- **`horizontal_confidence_idw_<surface>.png`** — heatmap of `weight_combined` over the
  evaluation grid (`viridis_r`, scale fixed 0–1). Dark = high confidence (close to
  controls), light = low confidence.

- **`horizontal_confidence_rank_<surface>.png`** — all grid nodes' `weight_combined`
  values sorted in descending order and plotted as a curve. Useful to see, at a glance,
  what fraction of the surface's footprint sits above any given confidence threshold
  (e.g. "what % of the grid has confidence ≥ 0.5?").

- **`interactive_confidence_<surface>.html`** — Plotly map of `weight_combined` on top of
  an OpenStreetMap basemap (or satellite if `MAPBOX_TOKEN` is set), with gray contour
  lines ("isolines") at every 0.1 confidence level (0.0, 0.1, ... 1.0) and a toggleable
  legend. Pan/zoom/hover to inspect confidence at any location.

### 9.4 Vertical confidence

- **`vertical_confidence_grid_<surface>.csv`** — one row per evaluation-grid node.
  Columns:
  - `x`, `y` — grid-node coordinates (`EPSG:6707`).
  - `abs_delta_z` — IDW-interpolated `|z_checkpoint - z_surface|` (meters) at this node,
    i.e. the estimated absolute vertical error of the modeled surface here, based on
    nearby topo-intersection checkpoints.
  - `abs_delta_norm` — `abs_delta_z` min-max normalized and **inverted** to `[0, 1]`
    (`1 - (val - min)/(max - min)`), so **1 = best (smallest |ΔZ|)**, **0 = worst
    (largest |ΔZ|)** *for this surface's grid*. This is what's plotted/mapped below.
  - `lon`, `lat` — same point reprojected to `EPSG:4326`.

- **`vertical_confidence_grid_<surface>.png`** — heatmap of `abs_delta_norm` (`plasma`,
  fixed 0–1 scale). Bright = high vertical confidence (small modeled-vs-DEM residual near
  this location), dark = low vertical confidence.

- **`vertical_confidence_grid_<surface>.html`** — Plotly interactive map of
  `abs_delta_norm`, same basemap/isoline/legend conventions as §9.3's interactive map, but
  colored with `plasma` and labeled "normalized |dZ| (1 = best)".

  > Note: despite sharing the `vertical_confidence_grid` name with the CSV, the value
  > plotted in the PNG/HTML is the **normalized** (0–1, inverted) confidence
  > `abs_delta_norm`, not the raw `abs_delta_z` in meters — use the CSV if you need the
  > raw residual.

### 9.5 Combined confidence (horizontal + vertical)

- **`combined_confidence_grid_<surface>.csv`** — one row per evaluation-grid node.
  Columns:
  - `x`, `y`, `lon`, `lat` — as above.
  - `horizontal_weight` — copy of `weight_combined` from §9.3 (H).
  - `vertical_norm` — copy of `abs_delta_norm` from §9.4 (V).
  - `arithmetic_combined` = `0.5*H + 0.5*V`.
  - `geometric_combined` = `H^0.5 * V^0.5` (geometric mean).
  - `min_combined` = `min(H, V)` — **this is the headline metric** (the run scripts use
    `mode="min"`): a location is only as confident as its *weakest* dimension.
  - `has_vertical` — always `True` here (only written when both H and V exist).

- **`combined_confidence_min_<surface>.png`** / **`combined_confidence_min_<surface>.html`**
  — heatmap / interactive map of `min_combined` (`magma_r`, 0–1). This is the
  **primary "overall confidence" output** of the pipeline for this dataset.

- **`combined_confidence_geometric_<surface>.png`** — heatmap of `geometric_combined`
  (`cividis`, 0–1), saved alongside the selected `min` mode purely for visual comparison
  (no separate `.html` for this one).

### 9.6 Boundary overlap (model vs. mapped trace) — symmetric

- **`boundary_overlap_<surface>.csv`** — single-row summary:
  - `surface` — surface name.
  - `move_length_m` — total length (m) of this surface's dissolved
    `3D_Topo_Intersections*` line (A, the model-computed outcrop trace).
  - `gis_length_m` — total length (m) of this surface's dissolved, matched non-tectonic
    `Limiti_stratigrafici_carta_geologica_DEF` contact line (B, the independently mapped
    trace).
  - `overlap_pct_A_to_B` — % of A's length falling inside a 20 m buffer around B.
  - `overlap_pct_B_to_A` — % of B's length falling inside a 20 m buffer around A.
  - `overlap_pct_mean` = `(overlap_pct_A_to_B + overlap_pct_B_to_A) / 2` — **the headline
    number and official comparison value**.
  - `buffer_dist_m` — the buffer tolerance used (20 m), applied symmetrically.
  - `n_move_segments_merged` / `n_gis_segments_merged` — number of raw line segments
    dissolved into A / B (sanity check on the matching in §5.1).

- **`boundary_overlap_<surface>.png`** — map showing the mapped contact (orange line) and
  its 20 m buffer (shaded orange polygon), the topo-intersection trace (blue line) and its
  20 m buffer (shaded blue polygon), and all three overlap percentages in the title.

- **`boundary_overlap_summary.csv`** (project-level, one file per run) — concatenation of
  all per-surface `boundary_overlap_<surface>.csv` rows. This is the table reproduced in §6.

### 9.7 Fault validation — symmetric

- **`fault_validation_per_fault.csv`** (project-level) — one row per matched fault:
  - `fault_id` — e.g. `Faglia 8a`.
  - `move_length_m` / `gis_length_m` — dissolved MOVE / GIS fault line lengths.
  - `fault_overlap_A_to_B` / `fault_overlap_B_to_A` / `fault_overlap_mean` — same symmetric
    formula as §9.6, **`fault_overlap_mean` is the main validation parameter**.

- **`fault_validation_aggregate.csv`** (project-level, single row):
  - `n_faults_compared`, `mean_fault_overlap_mean` (mean of `fault_overlap_mean` across all
    matched faults), `buffer_dist_m`.

- **`fault_validation_map.png`** — GIS fault traces (solid) vs. MOVE fault traces (dashed),
  one color per fault id.

- **`fault_throw_impact_<surface>.csv`/`.png`/`.html`** — IDW proxy for fault-throw impact
  on each horizon's grid (independent of the overlap metrics above).

### 9.8 Acceptance classification

- **`acceptance_table.csv`** — one row per horizon: `surface`, `overlap_pct_mean`,
  `mean_unit_thickness_m`, `acceptance_class`, `n_vertical_checkpoints`.
  `acceptance_class` is derived from `overlap_pct_mean` alone: `Accepted` (≥70),
  `Conditionally accepted` (≥50), `Rejected` (<50), `No data` (NaN).
- **`validation_summary.csv`** — one-row model aggregate: `n_surfaces`, `n_accepted`,
  `n_conditionally_accepted`, `n_rejected`, `n_no_data`, `mean_overlap_pct`,
  `min_overlap_pct` (mean/min, across surfaces, of each surface's `overlap_pct_mean`).

## 10. Assumptions and limitations

- Nearest-distance IDW is a simple proxy; it does not model anisotropy, structural trends,
  or true vertical accuracy/uncertainty.
- Quality depends on correct CRS, complete control data, and sensible grid/line-sampling
  parameters (§8).
- The maps/topo/fault matching (`Base_di`/`Horizon`/`Nome_fagli` → short code) is purely
  string-based; if a future version of `Limiti_stratigrafici_carta_geologica_DEF` or
  `Faglie_carta_geologica_DEF` renames a key value, that surface's/fault's maps tier and
  overlap output will silently go empty.
- The boundary-overlap and fault-overlap metrics are 2D planar comparisons — Z is dropped
  from both linesets before buffering/length calculations, even though both source
  shapefiles carry Z.
- The bidirectional comparison reduces, but does not eliminate, sensitivity to how each
  source was digitized: `overlap_pct_mean` (and `fault_overlap_mean`) is now a symmetric
  average of the A→B and B→A directions, so it is no longer dominated by which line happens
  to be longer/shorter or more finely segmented — this directly supersedes the old
  one-directional metric, which measured only how much of the *topo* trace was covered by
  the *map* buffer and could read near 100% even when the mapped contact was much
  longer/shorter than the topo trace.
