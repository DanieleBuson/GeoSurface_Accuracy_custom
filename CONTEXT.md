# GeoSurface_Accuracy_custom — context

This repository applies the **GeoSurface_Accuracy** accuracy-evaluation method
(originally [BaterHub/GeoSurface_Accuracy](https://github.com/BaterHub/GeoSurface_Accuracy);
see [`THEORY.md`](./THEORY.md) for the full method, fully self-contained in this repo) to
a different dataset: a 7-surface 3D geological model of an area in Trentino, with
**no wells** and a different set of GIS inputs. All shared logic (readers, IDW, plotting)
lives in [`files_utils.py`](./files_utils.py) (vendored here so this folder is
self-contained); [`custom_utils.py`](./custom_utils.py) is a thin wrapper that re-exports
it; [`custom_validation.py`](./custom_validation.py) holds the dataset-specific
stratigraphic matching, symmetric boundary/fault-overlap metric, and acceptance logic.

See [`README.md`](./README.md) for setup/run instructions (local and Google Colab).

## Dataset

All inputs live in [`working_files_folder/`](./working_files_folder/). Both
`run_accuracy_original.py` and `run_accuracy_resampled.py` read the **same full original
horizon/fault surfaces** — confidence-map processing always runs on complete original
horizons, never on split/segmented/partial surfaces:

| File | Content |
| --- | --- |
| `GOCAD_ASCII_All.ts` (read from `original/`) | 7 geological surfaces (`TOP_ARV`, `TOP_DPR`, `TOP_LOP`, `TOP_RTZinf`, `TOP_FMZ`, `TOP_RTZsup`, `TOP_OSV`), 343,861 vertices total |
| `GOCAD_ASCII_Faults.ts` (read from `original/`) | Fault surfaces used for fault validation |
| `3D_SectionsGrid.shp` | Control sections — 22 generic 3D section-trace lines (field `Name`), used as a "sections" checkpoint source for **every** surface |
| `3D_Topo_Intersections.shp` / `3D_Topo_Intersections_rsmpl_0.1.shp` | Horizon-topography intersections generated by MOVE (7 / densified features), field `Horizon`. Geometry Z = DEM elevation. The `_rsmpl_0.1` variant is a densified version (0.1 m vertex step). |
| `3D_Topo_Intersections_Faults.shp` / `3D_Topo_Intersections_Faults_rsmpl_0.1.shp` | Fault-topography intersections generated by MOVE, field `Name` (e.g. `F8a`); densified variant for the resampled run |
| `Limiti_stratigrafici_carta_geologica_DEF.shp` | Stratigraphic boundaries mapped in GIS (`Base_di`, `Tipo`, `Affidabili`) — the **sole** source of stratigraphic-contact geometry and cartographic reliability in this pipeline |
| `Faglie_carta_geologica_DEF.shp` | Faults mapped in GIS (`Nome_fagli`, `Tipo_fagli`) — the **official GIS reference** for all fault validation |
| `Giaciture_FB.shp` | Structural field measurements (strikes/dips); `TIPO` identifies fault vs. other measurements, used in the qualitative fault-throw comparison |

`Limiti_CartaGeol.shp` / `Limiti_CartaGeol_rsmpl_0.1.shp` are **fully retired** — they are
not read anywhere in this pipeline. `Limiti_stratigrafici_carta_geologica_DEF.shp` replaces
them in every pipeline operation: confidence maps, boundary overlap, cartographic checks,
and statistics.

There are **no well shapefiles** in this dataset, so `use_wells=False` everywhere.

## CRS

All shapefiles and the `.ts` model use the same projected meters (RDN2008 / UTM zone 32N).
Both pipelines standardize every layer to **`EPSG:6707`** right after loading, via
`files_utils.standardize_crs` — including layers whose `.prj` originally declared a different
(but numerically identical) UTM32N authority code, and `3D_Topo_Intersections_Faults*.shp`,
which carries no recognized authority code at all. No layer is exempt.

Per the "Linee Guida" PDF (§5, production workflow step 1), the **national GeoIT3D
submission standard** re-projects models to **`EPSG:6708` (RDN2008 / UTM zone 33N)**.
This dataset is **not** reprojected here — this is a local accuracy test, not a
submission package — but anyone preparing a real GeoIT3D delivery from this model
would need that reprojection step.

## How THEORY.md concepts map onto this dataset

(See [`THEORY.md`](./THEORY.md) §5 for the full explanation; summary below.)

| THEORY.md concept | This dataset |
| --- | --- |
| Wells (`p=1`) | Not present — `use_wells=False` for all surfaces |
| Sections (`p=2`) | `3D_SectionsGrid.shp`, applied unfiltered to **all 7 surfaces** (it's a generic section grid, not surface-specific) |
| Maps (`p=3`) | `Limiti_stratigrafici_carta_geologica_DEF.shp`, non-tectonic contacts matched to a surface via `Base_di` → `STRAT_MAP` → `_surface_short_code(surface)`, each sampled point weighted by `Affidabili` reliability (`AFFIDABILI_WEIGHTS`) — the sole source of cartographic reliability |
| Vertical Z checkpoints | `3D_Topo_Intersections*.shp`, matched via `Horizon` → `_normalize_horizon_code`. Geometry Z is the DEM elevation at the modeled outcrop — the most direct ground truth available |
| Fault traces | Model: `GOCAD_ASCII_Faults.ts` + `3D_Topo_Intersections_Faults*.shp` (`Name`); GIS: `Faglie_carta_geologica_DEF.shp` (`Nome_fagli`) — the official GIS fault reference |
| Mapping CSVs (`surface_data_mapping.csv` / `surface_checkpoint_edges.csv`) | **Not used.** Maps/topo/fault matching is fully automatic via short codes and the STRAT_MAP translation table; there's nothing to configure per surface |

## Boundary-overlap and fault-overlap metric (symmetric, bidirectional)

In addition to the horizontal/vertical/combined confidence outputs described in
THEORY.md, this dataset lets us compare two **independent** representations of each
surface's outcrop trace, and analogously for each fault:

- **`3D_Topo_Intersections*`** (line A) — the trace **interpolated by the modeling
  software** (surface ∩ DEM).
- **`Limiti_stratigrafici_carta_geologica_DEF`** (line B, non-tectonic contacts only) — the
  trace **explicitly mapped in the field/GIS**, independent of the 3D model.

Before comparison, every segment belonging to the same horizon (or the same fault id) is
merged into a single geometry per side via dissolve + line-merge. The overlap is then
computed in **both directions**, using a fixed **50 m buffer**:

```
overlap_pct_A_to_B = 100 * length(intersection(A, buffer(B, 50 m))) / length(A)
overlap_pct_B_to_A = 100 * length(intersection(B, buffer(A, 50 m))) / length(B)
overlap_pct_mean   = (overlap_pct_A_to_B + overlap_pct_B_to_A) / 2   # official comparison value
```

The same formula, with the same 50 m buffer, validates faults: MOVE fault traces
(`3D_Topo_Intersections_Faults*`, dissolved by `Name`) against GIS-mapped faults
(`Faglie_carta_geologica_DEF`, dissolved by `Nome_fagli`), reported as
`fault_overlap_A_to_B` / `fault_overlap_B_to_A` / `fault_overlap_mean`. **No P95 or other
distance-based metric is used anywhere in this pipeline** — `*_overlap_mean` is the only
surviving comparison value for both horizons and faults.

A high mean % means the model's computed trace agrees with the independently mapped
geology within 50 m, symmetrically in both directions; a low mean % flags a real
divergence between the 3D model and the field mapping. Making the metric bidirectional
removes the old asymmetry, where one direction alone could read near 100% purely because
one line was much shorter than (and fully buffer-covered by) a much longer counterpart.

Regenerated headline numbers (identical between the original and resampled runs):
`overlap_pct_mean` ranges 65.0–89.1% across the 7 horizons (mean 78.4%, all 7 now have a
`Base_di` match — `TOP_DPR` included, unlike with the old `Limiti_CartaGeol.shp`);
`fault_overlap_mean` ranges 0.0–100.0% across the 12 matched faults (mean 82.3%), with
faults `4` and `6` flagging genuine model/GIS divergence in both directions at once. See
[THEORY.md §6](./THEORY.md#6-boundary-overlap-and-fault-overlap-results-50-m-buffer) for
the full per-surface/per-fault tables, or `boundary_overlap_summary.csv` /
`fault_validation_per_fault.csv` in `output_results_original/` and
`output_results_resampled/` directly.

## Relevant sections of the "Linee Guida" PDF

[`Petricca_etal2025_LineeGuida_StrutturaBD_Nazionale_ModGeo3D.pdf`](./Petricca_etal2025_LineeGuida_StrutturaBD_Nazionale_ModGeo3D.pdf)
(ISPRA, PNRR GeoSciencesIR) defines the national 3D geological model database
structure. The parts most relevant here:

- **§3.2, Tabella 3 (`evaluation_method`)** and **Tabella 4 (`observation_method`)**
  define controlled vocabularies for *how* a surface/contact was determined
  (compilation, direct observation, indirect method, inferred, surveyed; resistivity/
  seismic/borehole/outcrop surveys, etc.). The `Affidabili` field in
  `Limiti_stratigrafici_carta_geologica_DEF` (`Osservato`/`Dedotto`/`Ipotizzato`/`Coperto`)
  is this dataset's local analogue of that evaluation-method/reliability concept, and is
  what drives the maps-tier reliability weighting (`AFFIDABILI_WEIGHTS`).
- **§5 (production workflow)** specifies re-projection to `EPSG:6708` (RDN2008/UTM33N)
  for the national submission package — see the CRS note above.
- **§4.4 ("Tool a supporto della valutazione dell'accuratezza")** explicitly points
  at the `GeoSurface_Accuracy` tool (this repository) as ISPRA's reference
  implementation for evaluating the accuracy of 3D model surfaces before submission
  — i.e. the additions in this folder extend the very tool the guidelines describe.

## How to run

See [`README.md`](./README.md) for full setup instructions, including how to run this on
Google Colab. Short version, from a Python environment with
[`requirements.txt`](./requirements.txt) installed:

```bash
# Original (non-resampled) topo-intersections
python run_accuracy_original.py
# or: jupyter nbconvert --to notebook --execute --inplace GeoSurface_Accuracy_Custom_original.ipynb

# Resampled (0.1 m step) topo-intersections
python run_accuracy_resampled.py
# or: jupyter nbconvert --to notebook --execute --inplace GeoSurface_Accuracy_Custom_resampled.ipynb
```

- **Original** uses `3D_Topo_Intersections.shp` / `3D_Topo_Intersections_Faults.shp` →
  outputs in `output_results_original/`.
- **Resampled** uses `3D_Topo_Intersections_rsmpl_0.1.shp` /
  `3D_Topo_Intersections_Faults_rsmpl_0.1.shp` → outputs in `output_results_resampled/`.
  Both runs load the **same full original horizon/fault surfaces**; the denser vertex
  sampling gives many more vertical checkpoints (tens of thousands vs thousands per
  surface) but the same `overlap_pct_*`/`fault_overlap_*` and IDW results, which is a
  useful consistency check.

Both runs produce, per surface: `horizontal_confidence_grid_*.csv` (with populated
`dist_maps`/`weight_maps`), `vertical_confidence_grid_*.csv`,
`combined_confidence_grid_*.csv`, heatmaps/HTML maps, `boundary_overlap_<surface>.csv`/`.png`
(for surfaces with a `Base_di` match), plus project-level `boundary_overlap_summary.csv`,
`fault_validation_per_fault.csv`, `fault_validation_aggregate.csv`,
`fault_validation_map.png`, `fault_throw_comparison.csv`, `acceptance_table.csv`, and
`validation_summary.csv`.
